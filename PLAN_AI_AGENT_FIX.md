## ДИАГНОЗ: ЧТО СЕЙЧАС НЕ ТАК

Бот Алекс сказал: *«Мои доступы = НОЛЬ. Нет функций для чтения БД. Только статичный prompt + твои сообщения.»*

**Реальность:** Инфраструктура Tool Use **полностью реализована** (Фаза 10.1), но работает неэффективно из-за нескольких критических разрывов в цепочке. Бот — как хирург с полным набором инструментов в чемодане, но без понимания, что чемодан можно открыть.

### Корневые причины (5 штук)

| # | Проблема | Критичность | Где |
|---|----------|-------------|-----|
| 1 | **System prompt не знает про Tool Use** — ни `system_max.txt`, ни `system_light.txt` не содержат ни слова о доступных инструментах. Claude не знает, что может вызывать функции | 🔴 КРИТИЧЕСКАЯ | `ai/prompts/system_max.txt`, `system_light.txt` |
| 2 | **L0-контекст не включает текущий вес** — `_build_l0_card()` показывает возраст, рост, fitness score, но НЕ последний вес из `metrics`. Бот буквально говорит «Вес: пустое поле» | 🔴 КРИТИЧЕСКАЯ | `ai/context_builder.py` → `_build_l0_card()` |
| 3 | **L0-контекст не включает место тренировок и оборудование** — поле `training_location` есть в БД, сохраняется при онбординге, но НЕ передаётся в промпт | 🟡 ВЫСОКАЯ | `ai/context_builder.py` → `_build_l0_card()` |
| 4 | **Regex-парсеры дублируют Tool Use** — `_process_user_input()` сначала прогоняет regex (строки 411-425 в handlers.py), потом запускает агентный цикл. Данные могут записаться дважды | 🟡 ВЫСОКАЯ | `bot/handlers.py` → `_process_user_input()` |
| 5 | **MAX_TOKENS = 1000 слишком мало** — контекст занимает ~800-1200 токенов, для ответа остаётся 0-200. Claude не может полноценно рассуждать и вызывать tools | 🟡 ВЫСОКАЯ | `config.py` |

## ПЛАН ДЕЙСТВИЙ

### ЭТАП 1. НАУЧИТЬ CLAUDE ПРО ИНСТРУМЕНТЫ (system prompt)
**Приоритет:** 🔴 P0 — без этого Tool Use не работает
**Файлы:** `ai/prompts/system_max.txt`, `ai/prompts/system_light.txt`
**Время:** 30 мин

#### Что сделать

В конец ОБОИХ промптов (system_max.txt и system_light.txt) добавить блок с описанием инструментов и правилами их использования. Claude должен знать:

**а) Какие инструменты доступны:**

```
═══ ИНСТРУМЕНТЫ (Tool Use) ═══
У тебя есть доступ к базе данных через инструменты. ВСЕГДА используй их для записи и чтения данных.

ЗАПИСЬ:
• save_workout — записать тренировку (тип, длительность, интенсивность, упражнения)
• save_metrics — записать метрики (вес, сон, энергия, настроение, вода, шаги)
• save_nutrition — записать питание (КБЖУ, описание еды)
• save_exercise_result — записать результат упражнения (подходы, повторы, вес, время)
• set_personal_record — зафиксировать новый PR

УПРАВЛЕНИЕ:
• update_athlete_card — обновить цель, уровень, дни, место, травмы, сезон
• award_xp — начислить XP за достижение
• save_episode — сохранить важный момент в долгосрочную память

ЧТЕНИЕ:
• get_weekly_stats — получить подробную статистику за N дней из БД
```

**б) Когда вызывать:**

```
═══ ПРАВИЛА ИСПОЛЬЗОВАНИЯ ═══
1. Когда пользователь сообщает о тренировке → save_workout + save_exercise_result (для каждого упражнения)
2. Когда сообщает вес/сон/энергию/настроение → save_metrics
3. Когда сообщает о еде/КБЖУ → save_nutrition
4. Когда спрашивает о прогрессе/статистике → get_weekly_stats
5. Когда меняет цели/предпочтения → update_athlete_card
6. Когда побит рекорд → set_personal_record + award_xp(200)
7. Важные моменты (инсайты, достижения, смена цели) → save_episode

ВАЖНО: НЕ спрашивай пользователя «хочешь записать?» — записывай автоматически.
ВАЖНО: Данные в разделе «ПАМЯТЬ О ПОЛЬЗОВАТЕЛЕ» — это снимок БД. Ты можешь получить более свежие данные через get_weekly_stats.
```

**в) Поведенческие инструкции:**

```
═══ КАК ТЫ РАБОТАЕШЬ ═══
Ты — AI-агент с доступом к базе данных. Ты НЕ просто чат-бот.
• Ты видишь полный профиль атлета (передан ниже в разделе ПАМЯТЬ)
• Ты можешь сохранять все данные через инструменты
• Ты можешь запрашивать детальную статистику
• Каждое сообщение пользователя анализируй: есть ли данные для записи?
```

#### Почему это критично

Anthropic API передаёт tools как отдельный параметр — Claude видит их технически, но без инструкции в system prompt он не знает *когда* и *зачем* их использовать. Это как дать человеку пульт от ТВ без объяснения, что такое кнопки.

### ЭТАП 3. УБРАТЬ ДУБЛИРОВАНИЕ REGEX-ПАРСЕРОВ
**Приоритет:** 🟡 P1 — двойная запись данных
**Файлы:** `bot/handlers.py` → `_process_user_input()`
**Время:** 15 мин

#### Что сделать

В `_process_user_input()` (строка 392) regex-парсеры на строках 411-425 выполняются ПЕРЕД agent loop. Это приводит к проблемам:

- Пользователь пишет «потренировался 45 минут, жим 3x10 80кг»
- Regex парсит и записывает в БД (грубый парсинг, теряет детали)
- Потом Claude вызывает `save_workout` + `save_exercise_result` через Tool Use (точный парсинг)
- Результат: дублирование записей

**Решение — отключить regex, оставить только Tool Use:**

```python
async def _process_user_input(tg_id, text, update):
    # 1. Сохраняем входящее сообщение
    save_user_message(tg_id, text)

    # 2. Regex-парсеры ОТКЛЮЧЕНЫ — Tool Use полностью заменяет их (Фаза 10.1)
    # Оставлены как комментарий на случай fallback:
    # if is_workout_report(text): ...
    # if is_metrics_report(text): ...
    # if is_nutrition_report(text): ...

    # 3. Авто-суммаризация
    compress_result = await maybe_compress_context(tg_id)
    ...

    # 4. Генерация через агентный цикл
    ...
```

#### Альтернатива (безопасная)

Если хочется оставить regex как safety net — добавить дедупликацию: перед regex-записью проверять, нет ли уже записи за сегодня. Но лучше просто отключить — Tool Use надёжнее.

### ЭТАП 4. УВЕЛИЧИТЬ MAX_TOKENS
**Приоритет:** 🟡 P1 — Claude не хватает места для рассуждений и tool calls
**Файлы:** `config.py`
**Время:** 2 мин

#### Что сделать

```python
# Было:
MAX_TOKENS = 1000

# Стало:
MAX_TOKENS = 1500
```

#### Почему

MAX_TOKENS — это лимит на **ответ** Claude (output), а не на контекст. Контекст (system + memory + history + tools) передаётся отдельно. При 1000 токенах Claude:
- Должен сгенерировать tool_use JSON (~100-200 токенов за каждый tool call)
- Потом сгенерировать финальный текст (~200-400 токенов)
- Итого нужно ~400-700 токенов минимум

С 1500 будет комфортный запас для цепочек из 2-3 tool calls + развёрнутый ответ.

**Стоимость:** Sonnet 4 = $3/$15 за 1M input/output. Увеличение на 500 output-токенов ≈ +$0.0075 за сообщение. Несущественно.

### ЭТАП 7. ДОБАВИТЬ TOOL USE В SCHEDULED MESSAGES
**Приоритет:** 🟢 P2 — утренние/вечерние чек-ины пока без Tool Use
**Файлы:** `ai/client.py` → `generate_checkin_response()`, `generate_scheduled_message()`
**Время:** 1 час

#### Проблема

Сейчас запланированные сообщения (утренний/дневной/вечерний чек-ин) используют `ask()` — синхронный клиент **без Tool Use**. Это значит:
- Утренний чек-ин не может записать метрики
- Вечерний отчёт не может сохранить итоги дня
- Только свободный диалог использует агентный цикл

#### Решение

Создать `generate_scheduled_agent_response()` — асинхронный аналог `generate_agent_response()` но для scheduler context:

```python
async def generate_scheduled_agent_response(
    bot, chat_id, system, prompt, tg_id, tools=None
) -> str:
    """Агентный цикл для scheduled сообщений (утро/вечер)."""
    if tools is None:
        from ai.tools import ALL_TOOLS
        tools = ALL_TOOLS
    # ... аналогично generate_agent_response, но с prompt вместо history
```

Затем в `scheduler/logic.py` заменить `ask()` на новую функцию.
